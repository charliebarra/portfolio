Flagship case study • 2025

Meteor Mayhem

An original multiplayer strategy card game about resource collection, ship upgrades, disruption, probability, and player decision-making.

RoleLead Designer & Creator
FormatStrategy Card Game
StatusPlayable Prototype
FocusBalance + Systems

Design challenge

Can a card game feel strategic, unpredictable, and fair?

I wanted to design a game where players had to make real decisions: when to collect resources, when to upgrade, when to disrupt someone else, and when to protect their own strategy. The hard part was making the game unpredictable without making it feel random or unfair.

Design goals

Meaningful choices

Players should have more than one reasonable path to winning.

Risk vs. reward

Mining stronger resources should feel tempting, but not automatic.

Player interaction

Events and disasters should create tension without ruining the game.

Replayability

The system should create different outcomes depending on card draw and player decisions.

Readable rules

Players should understand the cards quickly enough to focus on strategy.

Balance

No single strategy should feel obviously dominant every time.

System breakdown

Meteor Mayhem is built from interacting systems.

Resource System

Players collect resources with different values, making timing and risk important.

Upgrade System

Ship upgrades change capacity, mining strength, and long-term strategy.

Tool System

Tools give players tactical options like protection, stealing, recycling, and extra actions.

Event System

Events disrupt the game state through swaps, trades, blockades, and surprise effects.

Disaster System

Hazards add pressure and force players to adapt instead of following one perfect plan.

Scoring System

The winner is determined by total resource value, not just total cards collected.

Card design + UX

Cards had to be readable during play.

I used color, icons, short text, and consistent layouts so players could quickly identify card types and understand effects without stopping the game every turn.

Balancing

Balance was part math, part player psychology.

I modeled win probabilities based on deck composition and draw outcomes, then used playtesting to see whether the game felt balanced in practice. Some mechanics looked fair on paper but felt frustrating when real players used them.

Testing + iteration

Players never behave exactly how you expect.

Watching people play helped me find confusing rules, overpowered effects, pacing problems, and moments where the game became more interesting than I expected.

Reflection

What this project taught me about game design.

Iteration beats the first idea.

The game improved most when I was willing to change or remove mechanics that were not working.

Balance is felt by players.

A card can look mathematically fair but still feel annoying or too punishing during play.

Feedback creates better systems.

Testing with other players showed me interactions I never would have found alone.
